Guarding the audio thread

The worst-case rig proved that one stray operation — a single memory allocation — is invisible in the average and fatal in the tail. So how do we make sure it never sneaks in? Not willpower. Defense in depth: a few habits, plus an automatic tripwire. Here's the playbook, in annotated Rust.

The rule, and why

The audio thread runs under a hard 5.3 ms deadline, so every operation it performs must have a bounded, predictable cost. Anything whose worst case is unbounded — could ask the operating system for something, could wait on another thread — is banned. Concretely, a function on the audio thread must contain none of:

  • ✗ allocation — vec!, Box::new, String, format!, .collect()
  • ✗ deallocation — dropping any heap value (that's a hidden free())
  • ✗ blocking — Mutex::lock, RwLock, channels that may wait or allocate
  • ✗ I/O & syscalls — println!, files, sockets, the clock
  • ✗ unbounded work — HashMap growth, sorting unbounded input, unbounded loops
  • ✓ allowed: arithmetic, fixed slices, atomics, lock-free queues, preallocated pools

Every technique below is just a way of upholding one of those lines — and the last one is how we enforce them automatically instead of hoping.

1Allocate once, then only reuse

The single most common mistake: a scratch buffer created inside the render function. It allocates and frees on every block — two unbounded stalls, 188 times a second. The fix is to own that memory in the node and allocate it once, at setup, off the audio thread.

FORBIDDEN — allocates & frees every block
fn process(&mut self, out: &mut [f32]) {
    let mut scratch = vec![0.0; out.len()]; // ✗ a fresh heap buffer, every block
    mix_into(&mut scratch, out);
} // ← scratch is dropped here too: a free() on the hot path
REAL-TIME SAFE — borrow memory you already own
struct Reverb { scratch: Vec<f32> }   // allocated once in Reverb::new(block_size)

fn process(&mut self, out: &mut [f32]) {
    let scratch = &mut self.scratch; // ✓ just a borrow — zero allocation
    mix_into(scratch, out);
}

For things that come and go — synth voices, active notes — don't allocate per note. Keep a fixed pool (a Vec with reserved capacity, or a fixed array) and reuse slots. The pool is sized for the worst case up front; the audio thread only ever indexes into it.

2Talk through the membrane, never a lock

The UI thread changes parameters; the audio thread reads them. The tempting tool is a mutex — and it's exactly wrong, because .lock() can make the audio thread wait on the UI thread, blowing the deadline (a "priority inversion"). Instead: one value crosses as an atomic, a stream of events crosses on a lock-free queue.

FORBIDDEN — a lock can stall the audio thread
let p = self.params.lock().unwrap(); // ✗ .lock() may block on the UI thread
self.cutoff = p.cutoff;
REAL-TIME SAFE — non-blocking by construction
// one value → one atomic: a single, bounded CPU instruction
let cutoff = self.cutoff.load(Ordering::Relaxed); // ✓ never blocks

// a stream of events → a lock-free SPSC queue (e.g. the `rtrb` crate)
while let Ok(ev) = self.events.pop() { // ✓ drains what's queued, then returns
    self.apply(ev);
}

This is the membrane from the very first demo, grown up: the audio thread and the rest of the program meet only through primitives that cannot block it.

3Don't even free — ship the trash out

Subtle one: dropping a heap value (a Box, a Vec) calls free(), which is just as unbounded as malloc. So when a voice ends, you can't simply let its buffers drop on the audio thread. You move them onto a queue and let a housekeeping thread — which has no deadline — do the freeing.

REAL-TIME SAFE — hand-off now, free later
// audio thread: a voice finished — send its heap memory away, don't drop it here
if voice.done() {
    let _ = self.trash.push(voice.take_heap()); // ✓ move out, no free() on the hot path
}

// housekeeping thread (no deadline): the free() happens safely over here
while let Ok(garbage) = trash.pop() {
    drop(garbage); // ← free() runs off the audio thread
}

4Flush denormals to zero

A non-obvious tail-spiker: denormals — vanishingly small floating-point numbers that appear as a sound decays toward silence. On many CPUs the math on them runs 10–100× slower, so a fading reverb tail can quietly blow the budget. The fix is one setting, applied once when the audio thread starts.

// set the CPU's "flush-to-zero" mode once, at audio-thread start:
//   x86_64 : set FTZ + DAZ in the MXCSR register
//   aarch64: set the FZ bit in FPCR            (Apple Silicon)
// in practice: the `no_denormals` crate, or a few lines of core::arch.
fn on_audio_start() { no_denormals::flush_to_zero(); } // tiny near-zeros now cost nothing

5The tripwire: catch violations automatically

Habits slip. The real answer to "how do we prevent it" is to make a violation fail loudly, on its own. Install a custom global allocator that knows when we're on the audio thread (a thread-local flag) and panics the instant anything allocates there. Now any stray vec! — even one buried inside a dependency — blows up in your tests instead of clicking in a user's recording.

THE GUARD — a global allocator that trips on audio-thread allocation
use std::alloc::{GlobalAlloc, Layout, System};
use std::cell::Cell;

thread_local!(static ON_AUDIO: Cell<bool> = const { Cell::new(false) });

struct Tripwire; // wraps the normal system allocator
unsafe impl GlobalAlloc for Tripwire {
    unsafe fn alloc(&self, l: Layout) -> *mut u8 {
        ON_AUDIO.with(|on| if on.get() {
            panic!("allocation on the audio thread!"); // a bug — caught instantly
        });
        System.alloc(l)
    }
    unsafe fn dealloc(&self, p: *mut u8, l: Layout) { System.dealloc(p, l) }
}

#[global_allocator]
static GLOBAL: Tripwire = Tripwire;

Then flip the flag around the render callback:

ON_AUDIO.with(|on| on.set(true));
render_block(out);  // ✓ any stray alloc OR free in here now panics
ON_AUDIO.with(|on| on.set(false));

This is precisely what the assert_no_alloc crate does, more robustly — you enable it in debug and test builds and ship the release without the check. But a tripwire only catches what it can see. Locks that rarely contend, denormal storms, an occasional syscall — those slip past static guards. That's why the other half of prevention is empirical: run the worst-case rig and watch the tail. The allocator guard catches the obvious; the rig catches the rest.

Defense in depth

No single trick keeps the audio thread honest; the layers do it together: pre-allocate so there's nothing to allocate, talk lock-free so there's nothing to wait on, ship the trash out so there's nothing to free, flush denormals so the math stays cheap — and then a tripwire allocator plus the worst-case rig to catch whatever still slips through. Habits make it correct; the automatic checks keep it correct.

These are the same disciplines we'll lean on when we split the graph across CPU cores next — every worker thread plays by the exact same rules.
