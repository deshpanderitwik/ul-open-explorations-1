Structs & impl

A struct is a labelled box of data

Rust has no "classes." It splits the idea in two. First half: a struct just names a bundle of fields — the data that belongs together. Our oscillator only needs to remember two things between blocks, so that's all it holds:

pub struct SineOsc {
    phase: f32,        // where we are in the wave, 0.0..1.0
    sample_rate: f32,  // samples per second
}

That declaration is just a shape — a blueprint. When you make one, you get an instance: an actual box with those slots filled in. Picture it:

a SineOsc instance
phase : f320.0
sample_rate : f3248000.0

Notice the frequency is not in the box. That was a deliberate design choice we made back in the demo: frequency changes from outside every block, so it's passed into fill rather than stored. The struct holds only what must persist between calls.

Two shapes of struct

Our code uses both flavours. The difference is only how you name the fields — by label or by position.

named fields

struct SineOsc {
    phase: f32,
    sample_rate: f32,
}

Reach a field by name: self.phase. Best when fields mean different things.

tuple struct

struct AtomicF32(AtomicU32);

One unnamed field. Reach it by position: self.0. A thin wrapper around a single inner value — exactly our "smuggle an f32 through a u32" trick from Rung 1.

So when you saw self.0 in AtomicF32, it wasn't a typo — it's "field number zero," the lone AtomicU32 inside.

impl: where the behaviour lives

Second half of the split: the struct holds data, and a separate impl SineOsc { … } block holds its functions. Inside an impl there are two kinds of function, and the tell is whether the first parameter is self:

associated function

pub fn new(sample_rate: f32) -> Self {
    SineOsc { phase: 0.0, sample_rate }
}

No self → it isn't called on an instance; it makes one. Called as SineOsc::new(48000.0). Self is just a shortcut for the type's own name. Rust has no built-in constructor — new is pure convention.

method

pub fn fill(&mut self, buf: …, freq: f32) {
    // reads/writes self.phase
}

First param is &mut self → called on an instance: osc.fill(&mut buf, freq). It can touch the box's fields.

And here's the thing that ties Rung 2 and Rung 3 together — the one that makes Rust click. The dot is just sugar. A method call quietly passes the instance in as that first self parameter:

osc.fill(&mut buf, freq) is exactly the same as SineOsc::fill(&mut osc, &mut buf, freq) osc simply slides into the self slot

Why it matters. self was never magic — it's an ordinary first argument with a convenient shorthand. That's why &mut self from Rung 2 followed the same borrowing rules as any other parameter: it is just another parameter. Keep that in your pocket for Rung 4.

Tap new to construct the box, then fill repeatedly to advance its phase one block at a time.

Each fill advances phase by one block's worth (freq ÷ sample_rate × 256), wrapping at 1.0 — the exact arithmetic from the demo. new throws the box away and builds a fresh one with phase = 0. Two instances would each carry their own phase; that independence is the whole point of an instance.

The whole SineOsc, now readable

Three rungs in, every token below should land. Data in a struct; behaviour in an impl; new builds, fill mutates:

pub struct SineOsc {
    phase: f32,
    sample_rate: f32,
}

impl SineOsc {
    pub fn new(sample_rate: f32) -> Self {
        SineOsc { phase: 0.0, sample_rate }   // build the box, hand it back (no ;)
    }

    pub fn fill(&mut self, buffer: &mut [f32], freq: f32) {
        let phase_inc = freq / self.sample_rate;
        for sample in buffer.iter_mut() {
            *sample = (self.phase * TAU).sin();   // write one sample
            self.phase += phase_inc;            // advance — needs &mut self
            if self.phase >= 1.0 { self.phase -= 1.0; }
        }
    }
}

The only pieces still wearing a "trust me for now" label are &mut self and &mut [f32] — the borrows. That's the entire subject of the next rung.
