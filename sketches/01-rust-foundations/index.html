Every value has a type

In Rust a number is never just "a number." The language always knows how many bits it occupies and whether it can be negative. That precision is the reason Rust can compile to fast, predictable machine code — the kind a real-time audio thread depends on. These are the three number types our demo actually uses:

f32
32-bit floating-point (can be fractional & negative)
Every audio sample and every frequency. A sample lives in −1.0…1.0.
u32
32-bit unsigned integer (whole, never negative)
The raw bits we smuggle a float through inside AtomicF32.
usize
the integer type for sizes & positions
BLOCK_SIZE, and any index into a buffer.

You don't always have to write the type — often the compiler infers it. In let phase_inc = freq / self.sample_rate; we never said f32, but since both sides are f32, the result must be too. Rust figures it out, then holds you to it forever.

Nothing changes unless you say so

This is the rule that surprises everyone coming from JavaScript or Python: in Rust a binding is immutable by default. let x = 5 means x can never be reassigned or mutated. To allow change, you opt in — visibly — with mut. Three flavours, all in our code:

🔒

let

Immutable. Fixed once set. The default, and most values stay this way.

✏️

let mut

Mutable — you asked for it. The buffer is mut because the oscillator writes samples into it every block.

🗿

const

A name baked in at compile time. BLOCK_SIZE is 256 forever, before the program even runs.

Here are the real lines, annotated. This is the audio thread setting itself up:

const BLOCK_SIZE: usize = 256;        // 🗿 fixed at compile time, never changes

let mut buffer = vec![0.0f32; BLOCK_SIZE];  // ✏️ 256 zeros; mut → we'll write into it
//      ▲        ▲
//      │        └ the literal 0.0 with its type stapled on: f32
//      └ "mut" = this is allowed to change

let phase_inc = freq / self.sample_rate;     // 🔒 immutable; type f32, inferred

Why it matters. Mutability being a visible, deliberate choice is the seed of everything Rust is famous for. Because the compiler knows exactly what can change and what can't, it can later prove that two threads aren't scribbling on the same memory at once — which is precisely the bug the membrane is designed to avoid. Rungs 4 & 5 cash this in.

See the thing the code makes

All that f32 / mut machinery exists to do one job: fill a buffer with a wave. Here is that buffer, drawn live. Drag the frequency slider — that's you being the "UI thread." The wave reads the slider on every frame and redraws — that's the audio thread's load() at the top of each block. Higher frequency packs more cycles into the same buffer: the buffer changes shape.

freq = 220 Hz cycles in this 256-sample block = 1.17 sample rate 48000 · block 256 = 5.33 ms
the bars are the terminal sparkline (8 levels: ▁▂▃▄▅▆▇█); the line is the true wave

This is the browser mirror of what cargo run --release --bin membrane printed: a frequency moving across a one-number membrane, and a buffer visibly changing shape on the other side. Same idea, same math — just pixels instead of block glyphs.
