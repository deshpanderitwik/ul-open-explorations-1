The shape of every function

Here is a real one from our code — sparkline, the helper that turns a buffer into the little ▁▂▇█ waveform. It's a good first example because it's a plain function: no self, no &mut yet. Every part has a job:

pub fn sparkline(buffer: &[f32], width: usize) -> String { … }
pub fn — "a public function follows"
sparkline — its name
buffer, width — parameter names
&[f32], usize — each param's type
-> String — the type it hands back

The rule for parameters is the same one from Rung 1, repeated: every input is name: Type. Reading left to right: a public function called sparkline takes a buffer and a width, and gives you back a String. The arrow -> always points at what comes out.

What comes out — and the no-semicolon trick

Two things about returns trip up newcomers. First: if there's no ->, the function returns nothing — Rust calls that (), "unit." Second, and this is the genuinely Rust-y part: the last line without a semicolon is the return value. No return keyword needed. A semicolon ends a statement; leaving it off turns the line into "the value this block evaluates to."

Both patterns live in AtomicF32, side by side:

// Returns a value. Note: no semicolon on the last line → that's what comes back.
pub fn load(&self) -> f32 {
    f32::from_bits(self.0.load(Ordering::Relaxed))   // ← no ; = return this
}

// No arrow → returns nothing (unit). Its job is the side effect: writing the value.
pub fn store(&self, value: f32) {
    self.0.store(value.to_bits(), Ordering::Relaxed);  // ← ends in ; = a statement, nothing returned
}

Why it matters. Rust is expression-oriented: almost everything evaluates to a value, so the "last expression is the result" rule shows up everywhere — in if, in blocks, in match arms. Once you stop scanning for a return keyword and start reading "what does this block evaluate to?", huge amounts of Rust suddenly parse.

The special first parameter: self

Some functions are attached to a type — those are methods, and their first parameter is self (the thing the method is called on, like osc.fill(...)). There are three forms, and the difference is exactly the mutability idea from Rung 1, applied to "myself." This is also your first glimpse of the & symbol — the doorway to ownership we open fully in Rung 4.

formmeansin our code
&self "lend me a read-only look at myself." Can't change anything. load(&self) — just reads the stored bits.
&mut self "lend me myself with permission to change my fields." fill(&mut self, …) — it advances the oscillator's phase.
self "give me the whole thing, I'm taking ownership." (Rare; none in our demo yet.) —

The & is the heart of it: it means borrow, not take. &self borrows to read, &mut self borrows to write. Hold that intuition — Rung 4 is entirely about why borrowing is the mechanism that keeps two threads from corrupting each other.

Notice there's no ->: fill returns nothing. Its entire purpose is the side effect — leaving new samples in the buffer the caller handed it. That's why the audio thread can call it every block and just read the buffer afterward.
